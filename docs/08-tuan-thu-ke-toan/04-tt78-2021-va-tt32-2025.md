# 04. Hóa đơn điện tử: TT78/2021 → TT32/2025

> Quy định về hóa đơn điện tử (HĐĐT) tại Việt Nam.

## 1. Tổng quan法规

| TT | Hiệu lực | Trạng thái |
|----|---------|-----------|
| TT32/2011 | 2011-2018 | Hết hiệu lực |
| TT78/2021 | 01/07/2022 → 31/05/2025 | Đã thay thế |
| **TT32/2025** | **01/06/2025 trở đi** | **Đang áp dụng** |

### 1.1. TT78/2021/TT-BTC

- Ban hành: 17/09/2021
- Hiệu lực: 01/07/2022
- Bắt buộc tất cả DN chuyển sang HĐĐT
- 7 loại HĐĐT: GTGT, BH, BTS, HĐĐT special, PXK, ...

### 1.2. TT32/2025/TT-BTC (thay thế TT78)

- Ban hành: 28/02/2025
- Hiệu lực: 01/06/2025
- Điểm mới chính:
  - Mở rộng ủy nhiệm lập HĐĐT
  - Quy định về HĐĐT khởi tạo từ máy tính tiền
  - Quản lý dữ liệu HĐĐT chặt hơn
  - Quy định về tem điện tử

## 2. Loại hóa đơn điện tử

### 2.1. Phân loại theo nghiệp vụ

| Mã loại | Tên | Mô tả |
|---------|-----|------|
| 01 | Hóa đơn GTGT | Bán HH-DV cho DN có MST, có VAT |
| 02 | Hóa đơn bán hàng | Bán HH-DV cho cá nhân không MST |
| 03 | Hóa đơn bán tài sản | Bán TS đã sử dụng |
| 04 | Hóa đơn điều chỉnh | Điều chỉnh HĐ đã phát hành |
| 05 | Hóa đơn thay thế | Thay thế HĐ đã phát hành |
| 06 | Hóa đơn xuất khẩu | Xuất khẩu HH-DV |
| 07 | Phiếu xuất kho | Xuất nội bộ |
| 08 | Hóa đơn khác | Vàng bạc, đá quý, ... |

### 2.2. Ký hiệu mẫu số

- 1 ký tự cho loại HĐ
- 6 chữ số cho mã số:
  - Ký tự 1: loại HĐ (1-GTGT, 2-BH, ...)
  - Ký tự 2: có MST / không MST
  - Ký tự 3-4: năm (24 = 2024, 25 = 2025)
  - Ký tự 5-6: máy tính tiền (00 = không, 01+ = có)

Ví dụ: `1C25TPE` = HĐ GTGT, có MST, năm 2025, máy tính tiền, ký hiệu riêng

### 2.3. Số hóa đơn

- 7 chữ số tự nhiên, bắt đầu từ 1
- Tăng dần, không trùng lặp
- Mỗi dải số do DN đăng ký với cơ quan thuế

## 3. Nhà cung cấp HĐĐT

Tại VN có 4 loại hình nhà cung cấp HĐĐT:

1. **Doanh nghiệp cung cấp** (BKAV, Viettel, MobiFone, VNPT, ...)
2. **Cơ quan thuế cung cấp** (TCT trực tiếp - miễn phí)
3. **Doanh nghiệp tự phát hành** (phải được TCT phê duyệt)
4. **Máy tính tiền kết nối TCT**

Phần mềm PMKetoan cần hỗ trợ ít nhất:
- Tích hợp BKAV API (phổ biến nhất ở SME)
- Pull hóa đơn đầu vào từ TCT (HDDT)

## 4. Workflow HĐĐT đầu ra

```
┌─────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ Lập hóa đơn │ →  │ Ký số (CKS)  │ →  │ Gửi TCT + KH │ →  │ Lưu trữ 2N  │
│ (trong PMK) │    |              │    |              |    |             |
└─────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                          ↑                                       ↓
                          │                                       │
                  ┌──────────────┐                       ┌──────────────┐
                  │ Chữ ký số   │                       │ Cơ quan thuế│
                  │ (USB Token  │                       │ (Tổng cục)  │
                  │  hoặc HSM)  │                       └──────────────┘
                  └──────────────┘
```

### 4.1. Chi tiết quy trình

1. **Lập hóa đơn** trong PMKetoan:
   - Đầy đủ thông tin: người mua, HHDV, đơn giá, VAT, tổng
   - Lưu trạng thái `pending`
2. **Ký số**:
   - USB Token (cho DN nhỏ): người dùng cắm USB → nhập PIN → ký
   - HSM (cho DN lớn): ký tự động qua API
3. **Gửi TCT**:
   - Push XML có chữ ký lên API của nhà cung cấp (BKAV)
   - Nhà cung cấp chuyển tiếp cho TCT
   - TCT xác nhận → trả Mã CQT, Số HĐĐT
4. **Gửi khách hàng**:
   - Email / Zalo / VNP / ứng dụng riêng
   - Định dạng PDF + XML
5. **Lưu trữ**:
   - 2 năm tại hệ thống DN
   - Vĩnh viễn tại TCT

## 5. Workflow HĐĐT đầu vào

```
┌──────────────────────────────────────────────────────────────┐
│  Nhà cung cấp → BKAV → TCT → Cổng TCT → API TCT → PMKetoan  │
└──────────────────────────────────────────────────────────────┘
```

### 5.1. Pull hóa đơn từ TCT

TCT cung cấp API để doanh nghiệp pull HĐĐT đầu vào:

```python
# apps/purchasing/services/tct_service.py

class TCTPullService:
    def __init__(self, company):
        self.tax_code = company.tax_code
        self.api_key = company.tct_api_key
    
    def pull_invoices(self, from_date, to_date):
        """
        Pull danh sách HĐĐT đầu vào từ cổng TCT
        """
        response = requests.get(
            "https://api.thuedientu.gdt.gov.vn/api/invoice/input",
            params={
                'tax_code': self.tax_code,
                'from_date': from_date,
                'to_date': to_date,
            },
            headers={'Authorization': f'Bearer {self.api_key}'}
        )
        return response.json()
    
    def get_invoice_detail(self, invoice_key):
        """Get full XML của một HĐĐT"""
        response = requests.get(
            f"https://api.thuedientu.gdt.gov.vn/api/invoice/{invoice_key}",
            headers={'Authorization': f'Bearer {self.api_key}'}
        )
        return response.content  # XML
```

### 5.2. Match với phiếu nhập mua

```
1. Pull HĐĐT từ TCT → bảng input_invoice (status=pending)
2. Kế toán xem từng HĐ:
   - Nếu trùng với phiếu nhập mua đã có → match (status=matched)
   - Nếu chưa có phiếu nhập → tạo phiếu nhập mới từ HĐ
   - Nếu không dùng → exclude (status=excluded, lý do)
3. Sau kỳ: tất cả HĐ phải có status ≠ pending
```

## 6. Quy trình điều chỉnh / thay thế

### 6.1. Khi nào cần?

- Sai thông tin HĐ (số tiền, MST, tên KH, ...)
- Sau khi đã phát hành thành công
- Trong cùng kỳ kê khai

### 6.2. Hai phương án

**Điều chỉnh** (Adjustment invoice):
- Phát hành HĐ mới (mã loại 04) tham chiếu HĐ gốc
- Thay đổi số lượng, đơn giá, VAT (tăng hoặc giảm)
- HĐ gốc vẫn còn hiệu lực

**Thay thế** (Replacement invoice):
- Phát hành HĐ mới (mã loại 05) tham chiếu HĐ gốc
- HĐ gốc bị hủy (status=replaced)
- Chỉ áp dụng khi HĐ gốc sai nhiều / sai MST

## 7. Lưu trữ HĐĐT

### 7.1. Tại DN

- **Database**: XML raw + metadata (einvoice_id, no, date, ...)
- **File storage**: PDF + XML file trên disk hoặc S3
- **Backup**: full backup hàng ngày

### 7.2. Tại TCT

- Lưu trữ vĩnh viễn
- DN có thể query mọi lúc

## 8. Validation HĐĐT

### 8.1. Schema validation (XML)

```python
# apps/sales/services/einvoice_validator.py
from lxml import etree

def validate_einvoice_xml(xml_str):
    schema = etree.XMLSchema(etree.parse('schemas/einvoice.xsd'))
    try:
        schema.assertValid(etree.fromstring(xml_str))
        return True
    except etree.DocumentInvalid as e:
        raise EinvoiceValidationError(str(e))
```

### 8.2. Business rules

- MST người mua hợp lệ (check checksum)
- Tổng tiền = tổng dòng chi tiết
- VAT rate phải thuộc {0, 5, 8, 10, -KT}
- Mã CQT hợp lệ
- Số HĐ không trùng lặp

### 9.3. Async với django-q2

```python
# apps/sales/tasks.py
from django_q.tasks import async_task
from apps.sales.models import SalesInvoice
from apps.sales.services import EinvoiceService


def issue_einvoice(invoice_id):
    """Background task: phát hành HĐĐT qua BKAV"""
    try:
        invoice = SalesInvoice.objects.get(pk=invoice_id)
        service = EinvoiceService(invoice.company)
        result = service.issue(invoice)
        invoice.einvoice_status = 'issued'
        invoice.einvoice_no = result.invoice_no
        invoice.save()
    except Exception as exc:
        # Log + retry (django-q2 hỗ trợ retry qua `max_attempts` config)
        import logging
        logging.exception(f"Failed to issue einvoice {invoice_id}: {exc}")
        raise


# Trigger từ view/service:
from django_q.tasks import async_task
async_task(
    'apps.sales.tasks.issue_einvoice',
    invoice.id,
    task_name=f'einvoice_{invoice.id}',
)
```

## 10. Quy định mới TT32/2025

### 10.1. Điểm mới ảnh hưởng phần mềm

1. **HĐĐT khởi tạo từ máy tính tiền**:
   - Quy định rõ cho hộ kinh doanh
   - Cần log transaction ID

2. **Quản lý dữ liệu chặt hơn**:
   - Bắt buộc hash dữ liệu
   - Báo cáo định kỳ

3. **Tem điện tử**:
   - Áp dụng cho một số HH đặc thù (rượu, bia, thuốc lá)
   - Phần mềm cần hỗ trợ in/embed tem

4. **Ủy nhiệm lập HĐ**:
   - Hộ kinh doanh có thể ủy quyền cho DN khác lập HĐ
   - Cần field `is_authorized_issuer` và `original_issuer_tax_code`

### 10.2. Migration TT78 → TT32

- Hệ thống đã chạy TT78 không cần thay đổi lớn
- Cập nhật version API endpoint
- Cập nhật schema XML (nếu có)
- Đảm bảo backward compatibility
