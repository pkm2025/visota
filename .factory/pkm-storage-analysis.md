# PKM Data Storage Architecture Analysis

> Generated by deep investigation of `apps/pkm/` in the Visota ERP codebase.
> Covers all storage layers, multi-tenant isolation, encryption, lifecycle,
> system-level sharing, client-side cache, and backup/recovery.

## 1. Personal vs Collective Data Boundary

### Per-user data (private to a single user within a company)

All of the following models extend `apps.core.managers.CompanyOwnedModel`,
which means each row carries both a `company_id` FK and a `user_id` FK. They
are always queried with both filters combined.

| Model | DB table | Per-user scope |
|---|---|---|
| `KnowledgeNote` | `pkm_knowledge_note` | `user_id` + `company_id` |
| `PKMDocument` | `pkm_document` | `user_id` + `company_id` (+ `is_system` flag, see §5) |
| `Embedding` | `pkm_embedding` | `user_id` + `company_id` |
| `QAHistory` | `pkm_qa_history` | `user_id` + `company_id` |
| `UserInteractionLog` | `pkm_user_interaction_log` | `user_id` (nullable) + `company_id` |
| `Tag` | `pkm_tag` | `user_id` + `company_id` (unique per `(user, company, name)`) |
| `UserLLMConfig` | `pkm_user_llm_config` | `user_id` + `company_id` (unique per `(user, company, provider)`) |

### Per-user with no explicit company scoping

| Model | DB table | Notes |
|---|---|---|
| `DocumentChunk` | `pkm_documentchunk` | **Not** a `CompanyOwnedModel`. Inherently isolated via the parent `document` FK (cascade), but no direct `company_id`/`user_id` columns. See §3 for implications. |

### Shared / collective data

1. **System regulation documents** — `PKMDocument` rows with `is_system=True`.
   Shared across tenants via the vector-search UNION clause (see §5).
2. **Role-template notes** — `KnowledgeNote` rows owned by a superuser /
   first user whose `title` starts with `[mẫu]` and whose `role_context`
   matches a Visota role code. Surfaced to any user with the matching role
   via `apps.pkm.services.role_templates.get_role_suggestions_queryset`.

### Cross-user knowledge sharing mechanisms

- **Role templates** — company-scoped (not tenant-spanning). Any user whose
  `UserCompanyRole.role__code` matches a template note's `role_context` sees
  it in their dashboard, regardless of who owns the note. Implemented in
  `apps/pkm/services/role_templates.py`.
- **System regulation corpus** — `is_system=True` documents are pulled into
  every tenant's vector search results via the `include_system=True` branch
  of `vector_store.search_similar`. This is the only true tenant-spanning
  shared-knowledge mechanism.
- **No user-to-user sharing** within a company. Notes, documents, Q&A
  history, and tags are strictly per-user. There is no share/collaborate
  feature.

## 2. Storage Layers

### 2.1 Database tables (MariaDB)

Engine: `django.db.backends.mysql` (MariaDB) with `charset=utf8mb4`,
`sql_mode='STRICT_TRANS_TABLES'`. Configured in
`config/settings/base.py`.

Tables (all created via 7 migrations):

- `pkm_tag` — id, name, color, company_id, user_id, timestamps. Unique on
  `(user, company, name)`. Index on `(user, company)`.
- `pkm_knowledge_note` — id, title, content (markdown, TEXT), role_context,
  is_pinned, company_id, user_id, timestamps. M2M `tags` via
  `pkm_knowledge_note_tags`. Indexes on `(user, company)` and
  `(user, company, is_pinned)`.
- `pkm_user_llm_config` — id, provider, `api_key_encrypted` (TEXT),
  api_base, default_model, default_embedding_model, is_active, company_id,
  user_id, timestamps. Unique on `(user, company, provider)`. Indexes on
  `(user, company)` and `(user, company, is_active)`.
- `pkm_document` — id, title, file (FileField `pkm/docs/%Y/%m/`), file_type,
  file_size, status, checksum (SHA-256), error_message, `is_system`,
  company_id, user_id, timestamps. Indexes on `(user, company)`,
  `(user, company, status)`, `checksum`, and `is_system`.
- `pkm_documentchunk` — id, document_id, chunk_index, content (TEXT),
  token_count, created_at. Indexes on `document` and
  `(document, chunk_index)`. **No `company_id` / `user_id` columns.**
- `pkm_embedding` — id, chunk_id, user_id, company_id, content (cached
  text), model_name, `embedding VECTOR(1536)`, created_at. Indexes on
  `(user, company)` and `chunk`.
- `pkm_qa_history` — id, user_id, company_id, question, answer, sources
  (JSON), context_used (JSON), interaction_context (TEXT), created_at.
  Indexes on `(user, company)` and `(user, company, -created_at)`.
- `pkm_user_interaction_log` — id, user_id (nullable), company_id,
  interaction_type, module, entity_type, entity_id, metadata (JSON),
  created_at. Indexes on `(user, company)` and
  `(user, company, -created_at)`.
- M2M through table `pkm_knowledge_note_tags` (note_id, tag_id).

### 2.2 Vector storage

- **Engine**: MariaDB native `VECTOR(1536)` column on `pkm_embedding.embedding`.
- **Dimensions**: fixed at 1536 (matches OpenAI `text-embedding-3-small`).
- **Index**: HNSW vector index created via raw SQL in migration `0002`:
  ```sql
  CREATE VECTOR INDEX pkm_embedding_vec_idx
  ON pkm_embedding (embedding) M=8 DISTANCE=cosine;
  ```
- **Custom field**: `apps/pkm/fields.py::VectorField` declares
  `db_type = "VECTOR(N)"` so Django migrations manage the column. It is
  `editable=False`, `default=None`, and is **not writable via the ORM**.
- **Reads/Writes**: all vector I/O goes through raw SQL in
  `apps/pkm/services/vector_store.py`:
  - `store_embedding(...)` — `INSERT ... VEC_FromText(%s)` (vectors
    serialized via `json.dumps(list_of_floats)`).
  - `search_similar(...)` — `VEC_DISTANCE_COSINE(...) ORDER BY ... LIMIT N`.
  - `delete_embeddings(...)` — raw `DELETE` by `chunk_id` or `document_id`.

### 2.3 File storage

- **Backend**: `django.core.files.storage.FileSystemStorage` (default
  Django backend) configured in `config/settings/base.py::STORAGES`.
- **Location**: `MEDIA_ROOT = BASE_DIR / "media"` (dev) or
  `/app/media/` (prod, set in `config/settings/prod.py`).
- **Upload path**: `pkm/docs/%Y/%m/<filename>` (see `PKMDocument.file`
  `upload_to`).
- **No S3 / cloud object storage**. No `storages` package, no
  `boto3`/`django-storages` dependency, no `AWS_*` env vars in
  `.env.example`. Files live on the container/local filesystem.
- **WhiteNoise** is configured for *static* files
  (`CompressedManifestStaticFilesStorage`), not media. Media is served by
  Django in dev; in production behind Traefik there is no explicit reverse-
  proxy rule for `/media/` visible in this codebase.
- **Size limit**: 20 MB per upload (`MAX_FILE_SIZE` in `apps/pkm/api.py`).
- **Allowed types**: `pdf, docx, txt, md, xlsx`
  (`ALLOWED_FILE_TYPES`).
- **Dedup**: SHA-256 checksum stored on `PKMDocument.checksum`; duplicate
  uploads are flagged but still stored.

### 2.4 Cache

- **Django cache**: `DatabaseCache` backed by the `django_cache` table
  (`CACHES` in `base.py`). `TIMEOUT=3600` in dev/base, `300` in prod.
  No Redis. **The PKM module itself does NOT cache anything in the Django
  cache.** No `cache.get/set` calls exist in `apps/pkm/`. Caching is only
  used for sessions (`SESSION_ENGINE = cached_db`) and by other apps.
- **In-process LRU caches** (function-level, not data caches):
  - `apps/pkm/services/encryption_service.py::_derive_key` —
    `@lru_cache(maxsize=1)` caches the PBKDF2-derived Fernet key per
    process.
  - `apps/pkm/services/chunking_service.py` — `@lru_cache(maxsize=32)`
    caches tiktoken encoders per model name.

## 3. Multi-Tenant Isolation at the Storage Level

### How `company_id` + `user_id` scoping works

- The abstract base `apps.core.managers.CompanyOwnedModel` adds a non-null
  `company` FK (CASCADE) and exposes a `CompanyManager`/`CompanyQuerySet`
  with `.for_company(company_id)` helpers.
- Each PKM model *additionally* has its own `user` FK, so all PKM queries
  are double-scoped: `.filter(user=request.user, company=company)`.
- `request.current_company` is set by `apps.core.middleware.TenantMiddleware`
  from `session["current_company_id"]`, validated against
  `Company.objects.filter(id=..., is_active=True)`.
- The API layer (`apps/pkm/api.py`) consistently uses helper functions like
  `_get_scoped_queryset`, `_get_scoped_config`, and `_get_scoped_document`
  that apply both filters before returning any object. No PKM endpoint
  returns data without the `(user, company)` scope.
- Raw SQL in `vector_store.search_similar` explicitly filters
  `WHERE (e.user_id = %s AND e.company_id = %s)`.

### Potential cross-tenant leak risks

1. **`DocumentChunk` has no `company_id`/`user_id`.** This is safe *in
   practice* because chunks are only ever reached through their parent
   `PKMDocument` (which is scoped), and `on_delete=CASCADE` ensures no
   orphaned chunks survive. The vector search code joins
   `pkm_documentchunk -> pkm_document` to check `is_system`, and the
   embedding rows themselves carry `user_id` + `company_id`, so chunk
   content is only returned for embeddings that passed the tenant filter.
   **Risk**: if any future code queries `DocumentChunk` directly without
   going through `document` (and therefore `user`/`company`), it could
   leak content. There is no DB-level guard (no `company_id` column to
   constraint). Mitigation would be to add `company_id` to
   `DocumentChunk` or enforce manager-level filtering.

2. **`UserInteractionLog.user` is nullable** (migration `0006`). This
   allows system/automated business events to be logged without a user.
   The `company_id` is still required, so cross-tenant leakage is still
   bounded by company; however, queries that filter only on `user_id`
   could miss system rows, and queries that drop the user filter and rely
   on company alone would expose all users' activity within a company.

3. **System documents** (`is_system=True`) are attached to *one* specific
   company (the first company, or a sentinel `SYSTEM` company created by
   the seed command). The vector search `include_system=True` branch
   deliberately returns these rows to *every* tenant via the `OR
   COALESCE(d.is_system, 0) = 1` clause, bypassing the `(user_id,
   company_id)` filter. This is intentional shared knowledge, not a leak,
   but it does mean system regulation content is readable by every
   tenant's users through the Q&A pipeline.

4. **Role-template notes** are shared within a company via
   `get_role_suggestions_queryset`, which unions the user's own notes with
   any note in the same company whose title starts with `[mẫu]`. This
   is intentional intra-company sharing, scoped by `company_id`.

### No row-level security / DB-enforced isolation

MariaDB does not enforce tenant isolation at the DB level. All isolation is
in application code. There are no SQL views, no generated-column tricks,
and no database-level policies. A bug in any queryset that drops the
`company` filter would leak across tenants.

## 4. Data Lifecycle

### Document deletion

Triggered by `DELETE /api/v1/pkm/documents/{id}/`
(`apps/pkm/api.py::delete_document`). Cleanup is explicit and ordered:

1. `rag_pipeline.delete_document_data(doc.id)` is called:
   - `vector_store.delete_embeddings(document_id=...)` — raw SQL
     `DELETE FROM pkm_embedding WHERE chunk_id IN (SELECT id FROM
     pkm_documentchunk WHERE document_id = %s)`.
   - `DocumentChunk.objects.filter(document_id=...).delete()` — ORM
     delete (cascade also catches any remaining embeddings).
2. `doc.file.delete(save=False)` — physical file removed from
   `MEDIA_ROOT/pkm/docs/YYYY/MM/`. Wrapped in `suppress(Exception)` so a
   missing file does not block deletion.
3. `doc.delete()` — the `PKMDocument` row is removed.

This is a clean cascade. **However**, embeddings deleted via the raw SQL
path do **not** trigger Django signals (none are registered on `Embedding`
anyway), and the raw SQL delete runs *before* the ORM cascade, so the
behaviour is deterministic.

### User deactivation

- There is **no PKM-specific handler** for user deactivation or deletion.
- All PKM models use `user = ForeignKey(..., on_delete=CASCADE)`, so
  *deleting* a `User` row will cascade and destroy all of their notes,
  documents, chunks, embeddings, Q&A history, tags, LLM configs, and
  interaction logs.
- *Deactivating* a user (`is_active=False`) does **nothing** to their PKM
  data. The data persists and remains queryable. There is no management
  command to purge a deactivated user's PKM data, and no signal listens
  for `is_active` flips. **This is a gap**: a deactivated user's notes,
  documents, and especially `UserLLMConfig` (with encrypted API keys)
  remain in the database indefinitely.
- The seeding commands (`seed_pkm_regulations`, `seed_pkm_templates`)
  select the owner via `User.objects.filter(is_superuser=True,
  is_active=True).order_by("id").first()`, so a deactivated superuser will
  simply be skipped, not trigger any cleanup.

### Data retention policy

- **There is no retention policy anywhere in `apps/pkm/`.** No management
  command, no celery/django-q scheduled task, no `cleanup`/`purge`/
  `retention`/`expir` references exist in the PKM codebase.
- `UserInteractionLog`, `QAHistory`, and `Embedding` rows accumulate
  indefinitely. The `get_context_summary` service only looks at the last
  24 hours (`DEFAULT_SUMMARY_HOURS = 24`), but old rows are never deleted.
- There is no TTL on the `pkm_embedding` table, no partitioning, no
  archival strategy. For active users this will grow unboundedly.

### Interaction log purging

- No automatic purge. `UserInteractionLog` has `ordering =
  ["-created_at"]` and indexes on `(user, company, -created_at)`, which
  suggests an intent to support "recent-only" queries, but the cleanup
  job does not exist.

## 5. System-Level (`is_system=True`) Data

### Regulation documents

Seeded by `python manage.py seed_pkm_regulations`
(`apps/pkm/management/commands/seed_pkm_regulations.py`). Content lives in
`apps/pkm/management/commands/_regulation_content.py` as inline markdown
tuples `(slug, title, body)` covering:

- TT58/2026/TT-BTC (DNSN accounting regime)
- PIT/BHXH/CIT/VAT rates (July 2026)
- TT133/2016 chart-of-accounts overview
- ND 254/2026 e-invoice basics

Properties:

- `is_system=True`, status forced to `processed`.
- `company` is **not nullable** on `PKMDocument`, so system docs are
  attached to `Company.objects.order_by("id").first()` (or a sentinel
  `SYSTEM` company is created on the fly with `code="SYSTEM"`). They are
  **not** `company=None`.
- `user` is the first active superuser (or first user overall).
- Title prefixed with `[system] reg:<slug> —` for idempotent re-runs.
- The markdown body is also written to a `.md` FileField
  (`pkm/docs/<slug>.md`) so the existing `doc_parser.extract_text` path
  can read it.
- Chunks are created directly (bypassing the LLM embedding step), so
  system docs are retrievable by text even before any embedding model is
  configured.

### How vector search handles system docs

`vector_store.search_similar(..., include_system=True)` runs a UNION-style
query:

```sql
WHERE (e.user_id = %s AND e.company_id = %s)
   OR COALESCE(d.is_system, 0) = 1
```

This pulls the user's own embeddings *plus* any embedding whose parent
document has `is_system=True`, regardless of which company owns that
system document. `qa_service.answer_question` always calls with
`include_system=True`, so every tenant sees the shared regulation corpus.

Additionally, `qa_service._boost_regulation_chunks` subtracts
`REGULATION_BOOST_DISTANCE_BONUS = 0.15` from the cosine distance of any
`is_system=True` chunk when the question contains accounting/tax keywords
(Vietnamese + English list in `ACCOUNTING_KEYWORDS`). This soft-boost
prioritises regulation chunks for regulation-flavoured questions.

### Role templates

Seeded by `python manage.py seed_pkm_templates`
(`apps/pkm/management/commands/seed_pkm_templates.py`). Content lives in
`apps/pkm/management/commands/_role_template_content.py` as
`(role_code, slug, title, body)` tuples for `accountant`, `sales`,
`hr_officer`, `viewer`.

Properties:

- Stored as `KnowledgeNote` rows (not a separate model).
- `is_pinned=True`, `role_context=<role_code>`, title prefixed
  `[mẫu] tpl:<slug> —`.
- Owned by the first superuser / first user, attached to the first
  company (or sentinel `SYSTEM` company).
- **Shared within a company only** (not cross-tenant). Surfaced by
  `apps/pkm/services/role_templates.get_role_template_queryset` /
  `get_role_suggestions_queryset`, which filter on
  `KnowledgeNote.objects.filter(company=company, role_context__in=codes,
  title__startswith="[mẫu]")`.
- This means the seeding command attaches templates to whichever company
  happens to be first, and only users in *that* company benefit. Other
  tenants get no role templates unless the seed command is re-run against
  their company. **This is a gap** in true cross-tenant template sharing.

## 6. Encryption

### API keys (Fernet)

Implemented in `apps/pkm/services/encryption_service.py`.

- **Algorithm**: Fernet (symmetric authenticated encryption: AES-128-CBC +
  HMAC-SHA256).
- **Key derivation**: PBKDF2HMAC with SHA-256, 600,000 iterations (OWASP
  2023 guidance), 32-byte output, then base64url-encoded for Fernet.
- **Source secret**: `django.conf.settings.SECRET_KEY` (the same Django
  secret used for session/signing). The salt is a fixed, non-secret hash:
  `hashlib.sha256(b"visota.pkm.encryption.v1").digest()`.
- **Caching**: the derived key is cached per-process via
  `@lru_cache(maxsize=1)` on `_derive_key()`, so the expensive KDF runs
  only once per worker.
- **Storage**: `UserLLMConfig.api_key_encrypted` is a `TextField`
  holding the Fernet token (url-safe base64). Each encryption uses a
  fresh IV + timestamp, so the same plaintext yields different ciphertexts.
- **Decryption**: `_get_api_key` in `llm_service.py` decrypts in-memory
  per LLM call. The decrypted key is passed to litellm via the `api_key`
  kwarg and never stored globally, logged, or serialized. On
  `InvalidToken` (tampered/expired ciphertext), the failure is logged as
  a warning and `None` is returned, which surfaces as a provider auth
  error.

**Security note**: because the Fernet key is derived from
`SECRET_KEY`, anyone with DB read access AND the Django `SECRET_KEY`
(e.g. via env var leak) can decrypt all stored API keys. Rotating
`SECRET_KEY` invalidates every encrypted key (no re-encryption helper
exists; this is a gap).

### Document files at rest

**Not encrypted.** Files are stored as-is on the local filesystem
(`MEDIA_ROOT/pkm/docs/YYYY/MM/`) via `FileSystemStorage`. There is no
field-level encryption, no filesystem-level encryption hook, and no
`encrypted_file_storage` backend. Anyone with filesystem access to the
media directory can read every uploaded document.

### Vector embeddings at rest

**Not encrypted.** The `VECTOR(1536)` column is plain MariaDB storage.
Database-level encryption (e.g. MariaDB's InnoDB tablespace encryption,
or `key_management`) is not configured anywhere in the codebase. The
chunk text cached in `Embedding.content` and `DocumentChunk.content` is
also plaintext.

## 7. Client-Side Storage

### Dexie.js / IndexedDB (`static/modern/js/pkm-cache.js`)

Database name: `pkm_cache` (Dexie v1 schema). Three object stores:

| Store | Key path | Contents |
|---|---|---|
| `drafts` | `++id` (auto-inc) | `title`, `content`, `updated_at` — note drafts composed offline |
| `qa_history` | `++id` (auto-inc) | `question`, `answer`, `created_at` — mirror of recent Q&A for offline review |
| `cached_results` | `query_hash` | `results`, `created_at` — cached search/RAG results keyed by query hash |

Plus a second IndexedDB database `pkm_sync_queue` with an `outbox`
object store (auto-inc id) used as the Background Sync outbox. Each
entry holds `{url, method, headers, body, timestamp}` for replay.

API exposed on `window.PKMCache`:
`saveDraft`, `getDrafts`, `deleteDraft`, `getDraftByTitle`,
`saveQAHistory`, `getQAHistory`,
`cacheResults`, `getCachedResults`, `clearAll`,
`queueForSync`, `getQueuedCount`, `getQueuedRequests`,
`triggerSync`, `onSyncMessage`.

**No TTL / eviction logic** — drafts, Q&A cache, and cached results
persist in IndexedDB until `clearAll()` is called explicitly. No
automatic expiry is implemented client-side.

### Service worker (`static/sw.js`)

Cache version: `visota-v5`. Four caches:

- `visota-v5-static` — static assets (cache-first).
- `visota-v5-pages` — generic HTML pages (network-first).
- `visota-v5-pkm-pages` — PKM HTML pages
  (`/modern/knowledge/*`, stale-while-revalidate).

**Precached PKM assets** (on install):

- `/static/vendor/js/dexie.min.js`
- `/static/modern/js/pkm-cache.js`
- `/offline/`

**Precached PKM pages** (on install, for offline navigation):

- `/modern/knowledge/`
- `/modern/knowledge/notes/`
- `/modern/knowledge/qa/`
- `/modern/knowledge/settings/`

**Routing**:

- `GET /static/*` and asset extensions → cache-first.
- `GET /modern/knowledge/*` (HTML) → stale-while-revalidate into
  `pkm-pages` cache.
- `POST/PUT /api/v1/pkm/notes*` when offline → queued into the
  `pkm_sync_queue` outbox, returns HTTP 202, registers a Background Sync
  (`sync` tag `pkm-draft-sync`).
- Other HTML → network-first with cache fallback.
- Other JSON XHR → network-first.

### Offline draft persistence

1. User composes a note offline. `PKMCache.saveDraft(title, content)`
   writes to the `drafts` store in `pkm_cache`.
2. On submit, the page attempts the POST. If the SW intercepts an offline
   failure, `queueDraftRequest` stores the serialized request in the
   `pkm_sync_queue.outbox` IndexedDB store and returns 202.
3. When connectivity returns, the browser fires the `sync` event with
   tag `pkm-draft-sync`. The SW's `replayQueue()` replays each queued
   request with `credentials: "include"`, deletes successful ones, and
   broadcasts `pkm:sync-replayed` messages to clients. Failures stay
   queued for the next attempt.
4. The page can also force a manual replay via
   `PKMCache.triggerSync()` which posts `"PKM_SYNC_NOW"` to the SW.

**Gap**: only `/api/v1/pkm/notes*` POST/PUT are queued. Document uploads,
Q&A asks, and LLM config changes are not queued offline.

## 8. Backup / Recovery

### Built-in backup mechanism

- **None.** There is no PKM-specific backup command, no
  `dump_pkm_data`/`restore_pkm_data` management command, no scheduled
  `pg_dump`/`mariadb-dump` wrapper, and no reference to backups in
  `docker-compose.dev.yml`, `Makefile`, or the GitHub Actions workflows.
- Database-level backups (if any) are the operator's responsibility
  outside this repo. Nothing in the codebase automates them.
- The HNSW vector index is not portable across MariaDB instances via
  plain SQL dumps; restoring would require re-running the RAG pipeline
  (`reprocess_document`) for every document.

### User export of their knowledge base

- **No export endpoint exists.** Searching `apps/pkm/api.py` and
  `apps/ui_modern/views/` for any PKM export yields nothing. Users can
  list/read individual notes, documents, and Q&A history via the API, but
  there is no bulk-export (JSON, Markdown, ZIP) endpoint.
- **No import endpoint either.** Re-seeding is only possible via the two
  management commands (`seed_pkm_regulations`, `seed_pkm_templates`),
  which are admin-only and only seed system content, not user data.

### Disaster recovery implications

- A user who loses their device can re-authenticate and recover all
  server-side data (notes, documents, Q&A history) because everything is
  server-side.
- The client-side `pkm_cache` IndexedDB (drafts, cached Q&A) is
  per-browser and is **not** synced back to the server unless the user
  explicitly submits. Offline drafts that were never submitted exist
  only locally; losing the browser profile loses those drafts.
- If `SECRET_KEY` is rotated or lost, all `api_key_encrypted` values
  become undecryptable. There is no re-encryption utility.

## 9. Summary of Notable Gaps

| Area | Gap |
|---|---|
| Tenant isolation | `DocumentChunk` has no `company_id`/`user_id` columns; isolation depends entirely on going through the scoped `PKMDocument` parent. A direct query against `DocumentChunk` would leak content. |
| Lifecycle | No retention policy or purge job for `UserInteractionLog`, `QAHistory`, `Embedding`, or `DocumentChunk`. Data grows indefinitely. |
| Lifecycle | No handler for user *deactivation* (only cascading delete on user *deletion*). Deactivated users' data, including encrypted API keys, persists forever. |
| Sharing | Role templates are seeded against the first company only; other tenants do not automatically receive them. Cross-tenant template sharing requires re-running the seed command per company. |
| Sharing | No user-to-user sharing within a company (notes/documents/Q&A are strictly private). |
| Encryption | Document files on disk are unencrypted. Vector embeddings and cached chunk text are unencrypted. |
| Encryption | Fernet key is derived from `SECRET_KEY`; rotation invalidates all stored API keys with no re-encryption path. |
| Backup | No PKM-specific backup/restore command. No user-facing export of their knowledge base. |
| Offline | Only note POST/PUT is queued for background sync. Document uploads, Q&A asks, and config edits are dropped while offline. |
| Client cache | No TTL/eviction on `pkm_cache` IndexedDB stores; data persists until manual `clearAll()`. |
